# Phase 7 — Rule-Based Detection Policy

**Goal:** Give developers per-container control over what constitutes `FAILING`. The global Tier 2 threshold from Phase 6 becomes the fallback; named rules in `errorprobe.yaml` override it per container or glob.  
**Prerequisite:** Phase 6 complete.

**UT coverage requirement: ≥ 90% on all new packages and functions.**

---

## Design Overview

### Two rule types, two evaluation paths

Phase 6 introduced a single global evaluation path: every 30 s the `Tier2Evaluator` queries Loki for each `HAS_ERRORS` container and compares the error count against `cfg.Detection.Tier2.Threshold`. Phase 7 adds per-container rules that can override that threshold — and also adds a completely different kind of rule: **pattern rules**, which fire immediately when a specific string appears in any incoming log line.

These two rule types require two distinct evaluation paths:

| Rule type | Example | Evaluated where | Latency |
|---|---|---|---|
| **Threshold rule** — override `threshold` and/or `window` | `payments-api: threshold: 3` | `Tier2Evaluator.evaluate()` (periodic, every 30 s) | ≤ 30 s |
| **Pattern rule** — match a string in the message | `db-*: pattern: "connection refused"` | `Engine.ProcessBatch()` (real-time, on every inbound log event) | < 1 s |

Pattern rules must be real-time. Waiting 30 s to learn that a `"connection refused"` message appeared defeats the purpose. They are therefore checked inside `Engine.ProcessBatch`, which already runs synchronously on every inbound batch from Vector.

### How `Engine` gets access to rules

Currently `Engine` has no knowledge of `cfg` — it only knows about log events and state. `Tier2Evaluator` holds `cfg`. In Phase 7, `Engine.ProcessBatch` needs to check pattern rules, so it needs `cfg`.

Change: pass `cfg *config.Config` to `NewEngine`. The engine stores it as `e.cfg`. On a soft reload (`errorprobe reload` without container recreation), the reload path must update the engine's config pointer via a new `Engine.SetConfig(cfg *config.Config)` method so that new rules take effect immediately without restarting.

---

### `SetFailing` signature change

Currently: `SetFailing(name, fingerprint string, count int)`

Phase 7 adds `TriggeredBy string` to `ContainerHealth`. Rather than adding a 4th positional argument (fragile), the signature becomes:

```go
SetFailing(name string, opts FailingTransition) error

type FailingTransition struct {
    Fingerprint      string // dominant fingerprint (threshold rules); empty for pattern rules
    FingerprintCount int    // occurrence count; 0 for pattern rules
    TriggeredBy      string // e.g. "rule:payments-api" or "rule:db-* pattern:connection refused"
}
```

All existing callers in `tier2.go` update to pass a `FailingTransition` struct. The struct makes future extension (e.g. adding a reason code) non-breaking.

---

### Recovery for pattern-triggered FAILING

`evalFailing` in `Tier2Evaluator` currently calls `e.loki.CountErrors(name, window)` using the global threshold. In Phase 7 it calls `RuleFor(name, ...)` to get the per-container rule, then uses that rule's resolved `Threshold` and `Window` for the Loki count check.

For **pattern rules**: recovery fires when the Loki error count for the container drops below the rule's resolved threshold in the rule's resolved window. This is intentionally the same logic — we are not tracking whether the specific pattern is still occurring (Loki `CountErrors` counts all errors, not filtered by pattern). This is a deliberate V1 simplification: if the container has quieted down overall, it is considered recovered.

When `SetRecovered` is called it clears both `TriggeredBy` and `PatternMatchCounts` so future pattern matches start accumulating from zero.

---

### Rule resolution — `RuleFor`

```
config map key: "payments-api"  → exact match (priority 1)
config map key: "worker-*"      → glob match via path.Match (priority 2, first wins)
no match                        → returns rule built from cfg.Detection.Tier2 globals (priority 3)
```

Regardless of which level matched, the returned `DetectionRule` is always **fully populated**: any zero-value field (`threshold == 0`, `window == ""`) is filled in from the global `Tier2Config` before returning. Callers never need to check for zero values.

---

### `PatternMatchCounts` vs `Fingerprints`

These are two separate maps on `ContainerHealth` and serve different purposes:

| Field | Key | Purpose |
|---|---|---|
| `Fingerprints map[string]int` | SHA-256 fingerprint of normalised message | Used by `Tier2Evaluator` to find the dominant recurring error (threshold rules) |
| `PatternMatchCounts map[string]int` | Rule key from the config map (e.g. `"db-*"`) | Used by `Engine.ProcessBatch` to count pattern matches per rule (pattern rules) |

`PatternMatchCounts` accumulates across all received events and is reset to zero on recovery. It does not decay over time (unlike the Loki window query). It is therefore a one-way counter until the container recovers, which is appropriate: a zero-tolerance pattern should stay `FAILING` until something explicitly acknowledges it (recovery via rate drop) or the container restarts.

### Tier 2 — Rule resolution and pattern matching (depends on TRBP.1; independent of each other)

#### TRBP.2 — Implement `RuleFor` rule resolver in `health` package
File: `internal/health/rules.go` (new file)

- Signature: `RuleFor(containerName string, rules map[string]config.DetectionRule, globalDefaults config.Tier2Config) config.DetectionRule`
- Resolution order (see Design Overview above for rationale):
  1. Exact key match in `rules` map
  2. First glob match walking `rules` in insertion order using `path.Match`
  3. Fall back to a rule built from `globalDefaults`
- Before returning, fill any zero-value fields in the matched rule from `globalDefaults`:
  - If `rule.Threshold == 0` → set to `globalDefaults.Threshold`
  - If `rule.Window == ""` → set to `globalDefaults.Window`
  - `Pattern` is never defaulted — an empty pattern means "this is a threshold rule"
- Exact key match has absolute priority over globs even if a glob appears first in the map — iterate keys a second time for exact match before doing the glob scan
- Pure function — no I/O; fully unit-testable; no dependency on the engine or evaluator

#### TRBP.3 — Implement `MatchesPattern` function in `health` package
File: `internal/health/rules.go` (same new file as TRBP.2)

- Signature: `MatchesPattern(message, pattern string) bool`
- Match strategy — two modes, chosen by inspecting `pattern`:
  - Contains `*` or `?` → treat as a glob using `path.Match`; the whole message is the "name" and the pattern is applied as-is
  - Otherwise → `strings.Contains(strings.ToLower(message), strings.ToLower(pattern))` (case-insensitive substring)
- `pattern == ""` always returns `false` — an empty pattern means no pattern rule is active; this prevents accidental zero-tolerance triggers when a rule omits the `pattern` field
- Pure function — no I/O; deterministic; fully unit-testable

### Tier 3 — Evaluator integration (depends on TRBP.2, TRBP.3)

#### TRBP.4 — Extend `ContainerHealth`, `Engine`, and `Tier2Evaluator` with rule support

This task has three sub-parts that are implemented together because they form one coherent change to the FAILING state machine.

**Part A — Extend `ContainerHealth` struct** (`internal/health/health.go`)

Add two new fields to `ContainerHealth`:
```go
TriggeredBy       string         `json:"triggered_by,omitempty"`
PatternMatchCounts map[string]int `json:"pattern_match_counts,omitempty"`
```

Extend `Reset()` to clear both new fields so that a manual `status --reset` wipes all rule state:
```go
ch.TriggeredBy = ""
ch.PatternMatchCounts = nil
```

Extend `SetRecovered` in `engine.go` to also clear both new fields — recovery from FAILING must zero out `PatternMatchCounts` so future events start fresh.

**Part B — Extend `Engine.ProcessBatch` for real-time pattern matching** (`internal/health/engine.go`)

After the existing `RecordFingerprint` call in the `ProcessBatch` loop, add pattern rule evaluation:
```
rule := RuleFor(ev.Container, e.cfg.Detection.Rules, e.cfg.Detection.Tier2)
if rule.Pattern != "" {
    if MatchesPattern(ev.Message, rule.Pattern) {
        ch.PatternMatchCounts[ruleKey]++   // ruleKey is the matched config map key
        matchThreshold = max(rule.Threshold, 1)  // default to 1 if unset
        if ch.PatternMatchCounts[ruleKey] >= matchThreshold {
            // transition to FAILING immediately
            e.engine.SetFailing(name, FailingTransition{TriggeredBy: "rule:<key> pattern:<pattern>"})
        }
    }
}
```

`RuleFor` currently returns a `DetectionRule` but not the matched key. Add a second variant or change the return signature: `RuleForWithKey(containerName string, ...) (config.DetectionRule, string)` that returns both the resolved rule and the map key that matched (empty string on global fallback). The key is needed to populate `PatternMatchCounts` and `TriggeredBy`.

Pattern matching in `ProcessBatch` runs while the engine lock is held (same as the rest of the loop) — this is safe because `RuleFor` and `MatchesPattern` are pure functions with no I/O.

**Part C — Extend `SetFailing` signature** (`internal/health/engine.go`)

Change `SetFailing` from:
```go
func (e *Engine) SetFailing(name, fingerprint string, count int) error
```
to:
```go
type FailingTransition struct {
    Fingerprint      string // dominant fingerprint (set by threshold rules); empty for pattern rules
    FingerprintCount int    // occurrence count; 0 for pattern rules
    TriggeredBy      string // e.g. "rule:payments-api" or "rule:db-* pattern:connection refused"
}

func (e *Engine) SetFailing(name string, t FailingTransition) error
```

Inside `SetFailing`, set `ch.TriggeredBy = t.TriggeredBy` in addition to the existing fingerprint fields.

Update all callers in `tier2.go` (the `evalHasErrors` function) to construct a `FailingTransition` struct.

**Part D — Extend `Tier2Evaluator.evaluate` for per-container thresholds** (`internal/health/tier2.go`)

Currently `evaluate()` computes `window` and `threshold` once from global config and passes them to `evalHasErrors` and `evalFailing`. Replace this with per-container resolution:

```go
// Before (Phase 6):
window := parse(e.cfg.Detection.Tier2.Window)
threshold := e.cfg.Detection.Tier2.Threshold
e.evalHasErrors(ctx, name, ch, window, threshold)

// After (Phase 7):
rule, ruleKey := RuleForWithKey(name, e.cfg.Detection.Rules, e.cfg.Detection.Tier2)
window := parse(rule.Window)
threshold := rule.Threshold
e.evalHasErrors(ctx, name, ch, window, threshold, ruleKey)
```

Pass `ruleKey` through to `evalHasErrors` so it can include it in the `FailingTransition.TriggeredBy` field when calling `engine.SetFailing`.

`evalFailing` also calls `RuleForWithKey` so it uses the per-container window/threshold for the recovery check (instead of global values). This ensures a container with `threshold: 3, window: 1m` recovers when it drops below 3 errors per 1 minute, not below the global threshold.

**Pattern-triggered FAILING containers in `evalFailing`:** `evalFailing` is called for ALL `FAILING` containers regardless of what triggered them (threshold rule or pattern rule). The recovery check is the same in both cases: Loki error count below the resolved rule threshold in the resolved window. This is intentional — see Design Overview for the rationale.

## Manual Tests

Run after all tasks are complete:

1. **Global fallback unchanged** — remove `detection.rules` from `errorprobe.yaml`; emit 10 errors from a container; confirm it transitions to `FAILING` using global threshold. Existing Phase 6 behaviour must be unaffected.
2. **Stricter threshold rule** — set `payments-api: threshold: 3, window: 1m`; emit 3 errors from `payments-api` within 1 minute; confirm `FAILING` at count 3, not 10. A different container (no rule) must still use the global threshold.
3. **More tolerant threshold rule** — set `worker-1: threshold: 50`; emit 15 errors; confirm `worker-1` stays at `HAS_ERRORS` while a container using the global threshold would have transitioned.
4. **Zero-tolerance pattern rule** — set `db-main: pattern: "connection refused"`; emit one error containing `"connection refused"`; confirm `db-main` transitions to `FAILING` immediately on the first match.
5. **Pattern rule with threshold** — set `auth-service: pattern: "OOMKilled", threshold: 2`; emit one matching error; confirm no transition. Emit second matching error; confirm `FAILING`.
6. **`triggered by` in status** — with a FAILING container from a pattern rule, run `errorprobe status`; confirm `(triggered by: rule:db-main pattern:connection refused)` is present on the FAILING line.
7. **`triggered by` in watch** — with a FAILING container, open `errorprobe watch`, expand the FAILING row with `[e]`; confirm `Triggered by:` line is visible in the detail view.
8. **`triggered by` in JSON** — run `errorprobe status --json`; confirm `triggered_by` field is present and correct for FAILING containers; confirm it is `""` for OK and HAS_ERRORS containers.
9. **Recovery clears trigger** — container recovers from FAILING; confirm `TriggeredBy` is absent in `errorprobe status` output and cleared in `health.json`.
10. **Glob rule matching** — set `worker-*: threshold: 50`; start containers `worker-1` and `worker-2`; emit 15 errors from each; confirm both stay at `HAS_ERRORS` (glob applied to both).
11. **Exact key beats glob** — set both `worker-1: threshold: 3` and `worker-*: threshold: 50`; emit 5 errors from `worker-1`; confirm `worker-1` uses threshold 3 (exact match) not 50 (glob).
12. **`go test ./... -cover`** — all tests pass; coverage ≥ 90% on updated `internal/config` and `internal/health`.
